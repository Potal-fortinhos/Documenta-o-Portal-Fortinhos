1. Introdução e objetivo

Este documento de arquitetura tem como base o PDR · Levantamento de Requisitos (home.html). Descreve a estrutura de software do Portal do Aluno FUCAPI — incluindo decisões de estilo arquitetural, tecnologias, módulos e alocação dos requisitos funcionais/não funcionais. A arquitetura segue o padrão em camadas com API REST, priorizando escalabilidade, segurança (JWT, criptografia em repouso) e suporte a 800+ usuários ativos conforme RF e NF.

✔ baseado em RF001..RF006 ✔ NF001 a NF004 ✔ atores: aluno, professor, secretaria

Estilo arquitetural

Monolito modular API-first JWT

Comunicação: front-end consome endpoints JSON. Backend expõe rotas /api/v1/*. Futura escalabilidade horizontal via balanceador.

Padrões: MVC no backend, Repository pattern para abstração do BD, Front-end com componentes reutilizáveis (JS puro ou futuramente Vue).

Referência: requisitos não funcionais NF001, NF003, NF004

3. Metas e restrições arquiteturais

  • Segurança: Autenticação via JWT com expiração 30 min (NF001). Criptografia em repouso para dados sensíveis (NF002).
  • Desempenho: suportar 500 requisições simultâneas com resposta < 2s (NF003). Back-end em Node.js assíncrono.
  • Disponibilidade: responsividade e acessibilidade WCAG 2.1 AA (NF004).
  • Restrição tecnológica: front-end HTML5/CSS3/JS (conforme PDR), back-end Node.js, PostgreSQL.
  • Logs de auditoria: exigência do RF004 (alterações de notas registradas).

8. Segurança, desempenho e interoperabilidade

🔒 segurança em camadas

  • Autenticação JWT (assinatura HMAC-SHA256).
  • Dados sensíveis criptografados em repouso (AES-256).
  • Logs de alteração de notas (RF004) armazenados em tabela separada.
  • Controle de acesso por role (aluno, professor, secretaria).

⚡ desempenho (NF003)

  • Uso de cluster do Node.js (número de núcleos).
  • Cache Redis planejado para consultas de calendário/notas.
  • Consultas otimizadas com índices compostos (aluno_id + bimestre).
  • Compressão gzip nas respostas HTTP.

📌 Decisões arquiteturais futuras

Migração para microsserviços se houver expansão de domínio; API versionada desde o início (/v1). Mock de serviços externos (LDAP, e-mail) facilitam testes. Front-end preparado para evoluir para framework SPA.
